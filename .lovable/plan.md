

# Plan: Fix `_TMP_` Journal Entry IDs

## Problem

There are 14 journal entries in Supabase with corrupted IDs containing `_TMP_` prefixes (e.g., `_TMP__TMP__TMP__TMP_002-Q1-26`). These entries have **no corresponding lines in `journal_lines`**, which is why "no sale nada del asiento" -- the header exists but there's no detail data.

The `_TMP_` prefix doesn't exist in the current codebase. This was likely generated by a previous version of the chronological ID reassignment logic that used temporary IDs during renumbering but failed to complete (leaving orphaned headers without lines).

## Root Cause

The entries are orphaned headers in `journal_entries` with no matching rows in `journal_lines`. They cannot be recovered -- the line data is lost.

## Solution

Two changes:

### 1. Clean up corrupted data from Supabase
Delete the 14 orphaned `_TMP_` entries from `journal_entries` since they have no lines and cannot be displayed or used.

### 2. Add defensive filtering in `loadEntries` (data-adapter.ts)
After loading entries, filter out any that have zero lines to prevent empty/broken entries from appearing in the UI. This prevents future occurrences of similar issues.

### 3. Add defensive filtering in `JournalEntriesTable.tsx`
Skip rendering entries with empty `lines` arrays so they don't show as blank rows.

## Files to modify
- `src/accounting/data-adapter.ts` -- filter out entries with no lines in `SupaAdapter.loadEntries()`
- `src/components/journal/JournalEntriesTable.tsx` -- skip entries with `lines.length === 0`
- Database cleanup: delete orphaned `_TMP_` rows from `journal_entries`
